# Logging Contract: Execution Observability

**Feature**: 005-execution-observability  
**Date**: 2026-03-29  
**Version**: 1.0

## Contract Overview

This contract defines the logging interface and behavior for the execution observability feature. It specifies the format, content, and delivery guarantees for all execution logs generated by the system.

## Log Entry Contract

### Required Fields

Every execution log entry MUST contain the following fields:

```json
{
  "timestamp": "string (ISO-8601)",
  "executionPhase": "START|COMPLETION",
  "componentType": "CONTROLLER|SERVICE",
  "componentIdentifier": "string",
  "correlationId": "string",
  "executionStatus": "SUCCESS|FAILURE|ERROR",
  "threadName": "string",
  "mdcContext": {
    "correlationId": "string"
  }
}
```

### Conditional Fields

The following fields MUST be included when applicable:

```json
{
  "httpMethod": "string (CONTROLLER entries only)",
  "requestPath": "string (CONTROLLER entries only)",
  "executionDuration": "number (COMPLETION entries only)",
  "errorCode": "string (FAILURE entries only)"
}
```

### Field Specifications

#### timestamp
- **Format**: ISO-8601 UTC timestamp
- **Example**: "2026-03-29T10:45:00.123Z"
- **Requirement**: Must represent the exact time the log entry was created

#### executionPhase
- **Values**: "START" or "COMPLETION"
- **Requirement**: Indicates whether the log represents method start or completion

#### componentType
- **Values**: "CONTROLLER" or "SERVICE"
- **Requirement**: Indicates the architectural layer of the logged component

#### componentIdentifier
- **Format**: Fully qualified class and method name
- **Example**: "com.luscadevs.journeyorchestrator.application.service.JourneyInstanceService.createJourneyInstance"
- **Requirement**: Must uniquely identify the method being logged

#### correlationId
- **Format**: String identifier
- **Requirement**: Must be consistent across all log entries for a single request

#### executionStatus
- **Values**: "SUCCESS", "FAILURE", or "ERROR"
- **Requirement**: Indicates the outcome of method execution

#### executionDuration
- **Format**: Number representing milliseconds
- **Requirement**: Must be present for COMPLETION entries, must be non-negative

#### errorCode
- **Format**: Domain-specific error code
- **Requirement**: Must be present for FAILURE entries, must be defined in domain error taxonomy

## Log Format Contract

### JSON Structure
All execution logs MUST be formatted as valid JSON objects with the following structure:

```json
{
  "timestamp": "2026-03-29T10:45:00.123Z",
  "executionPhase": "COMPLETION",
  "componentType": "SERVICE",
  "componentIdentifier": "com.luscadevs.journeyorchestrator.application.service.JourneyInstanceService.createJourneyInstance",
  "correlationId": "req-12345-abcde",
  "executionDuration": 45,
  "executionStatus": "SUCCESS",
  "threadName": "http-nio-8080-exec-1",
  "mdcContext": {
    "correlationId": "req-12345-abcde",
    "httpMethod": "POST",
    "requestPath": "/api/journeys",
    "threadName": "http-nio-8080-exec-1"
  }
}
```

### Field Order
Fields MUST appear in the following order for consistent parsing:
1. timestamp
2. executionPhase
3. componentType
4. componentIdentifier
5. correlationId
6. httpMethod (if applicable)
7. requestPath (if applicable)
8. executionDuration (if applicable)
9. executionStatus
10. errorCode (if applicable)
11. threadName
12. mdcContext

## Delivery Guarantees

### Reliability
- **Success Rate**: 99.9% of log entries MUST be successfully generated and written
- **Failure Handling**: Logging failures MUST NOT impact primary business logic
- **Retry Logic**: Transient logging failures MUST be retried with exponential backoff

### Performance
- **Overhead**: Logging MUST add less than 2% latency to normal request processing
- **Timing Accuracy**: Execution duration MUST be accurate within ±5ms for operations >10ms
- **Throughput**: System MUST handle current request volume without logging bottlenecks

### Ordering
- **Request Ordering**: Log entries within a single request MUST be chronologically ordered
- **Cross-Request Ordering**: No ordering guarantees required across different requests

## MDC Context Contract

### Standard Context Fields
All log entries MUST include the following MDC context fields when available:

```json
{
  "mdcContext": {
    "correlationId": "string (required)",
    "httpMethod": "string (controller layer)",
    "requestPath": "string (controller layer)",
    "errorCode": "string (error conditions)",
    "threadName": "string (always included)"
  }
}
```

### Context Propagation Rules
1. **Initialization**: correlationId MUST be set at request start
2. **Propagation**: Context MUST be propagated through all synchronous calls
3. **Augmentation**: Additional context fields MAY be added during processing
4. **Cleanup**: Context MUST be cleared at request end

## Security Contract

### Data Exclusion
The following data MUST NOT appear in any execution logs:
- Method parameters or return values
- Request body content
- Authentication tokens or credentials
- Personal identifiable information (PII)
- Passwords, secrets, or sensitive configuration

### Data Sanitization
- **Automatic Exclusion**: Parameters with names containing "password", "token", "secret", "key", "credential" MUST be automatically excluded
- **Manual Exclusion**: Additional sensitive parameters MUST be explicitly excluded via configuration
- **Validation**: Automated scanning MUST verify no sensitive data leakage

## Integration Contract

### Component Interfaces

#### ExecutionLoggingAspect
```java
@Aspect
@Component
public class ExecutionLoggingAspect {
    // Intercepts @RestController methods
    @Around("@within(org.springframework.web.bind.annotation.RestController)")
    public Object logControllerExecution(ProceedingJoinPoint joinPoint) throws Throwable;
    
    // Intercepts @Service methods
    @Around("@within(org.springframework.stereotype.Service)")
    public Object logServiceExecution(ProceedingJoinPoint joinPoint) throws Throwable;
}
```

```java
@Component
public class MDCErrorEnhancer {
    // Enhances MDC with error codes during exception handling

```

## Testing Contract

### Unit Test Requirements
- **Aspect Testing**: All AOP aspects MUST have unit tests with mocked method calls
- **MDC Testing**: MDC context management MUST be tested for thread safety
- **Timing Testing**: Duration measurement accuracy MUST be verified
- **Format Testing**: Log output format MUST be validated against JSON schema

### Integration Test Requirements
- **End-to-End**: Complete request flows MUST generate expected log sequences
- **Performance**: Logging overhead MUST be measured under various load conditions
- **Error Scenarios**: Exception handling MUST generate appropriate error logs
- **Context Propagation**: MDC context MUST be consistent across all layers

### Contract Test Requirements
- **Schema Validation**: All log entries MUST validate against JSON schema
- **Field Completeness**: Required fields MUST be present in all scenarios
- **Data Integrity**: Field values MUST conform to specified formats
- **Security**: Sensitive data exclusion MUST be verified

## Monitoring Contract

### Health Metrics
- **Log Generation Rate**: Number of log entries generated per second
- **Log Success Rate**: Percentage of successfully written log entries
- **Average Log Size**: Average size of log entries in bytes
- **MDC Context Consistency**: Percentage of logs with complete MDC context

### Alerting Conditions
- **Log Success Rate < 99.9%**: Indicates logging system problems
- **Average Log Size > 2KB**: May indicate data leakage or formatting issues
- **Missing Correlation IDs**: Indicates MDC context problems
- **High Logging Overhead**: Performance degradation detected

## Versioning Contract

### Backward Compatibility
- **Field Addition**: New fields MAY be added without version change
- **Field Removal**: Fields MUST NOT be removed without major version change
- **Field Type Changes**: Field types MUST NOT be changed without major version change
- **Format Changes**: JSON format MUST remain stable within major version

### Deprecation Policy
- **Field Deprecation**: Deprecated fields MUST be supported for at least one minor version
- **Feature Deprecation**: Deprecated features MUST be supported for at least one major version
- **Migration Path**: Clear migration paths MUST be provided for breaking changes

## Compliance Contract

### Audit Requirements
- **Complete Coverage**: All controller and service methods MUST be logged
- **Immutable Records**: Log entries MUST be immutable once written
- **Tamper Detection**: Log integrity MUST be protected against tampering
- **Retention**: Logs MUST be retained according to organizational policies

### Regulatory Compliance
- **Data Protection**: Must comply with GDPR, CCPA, and other applicable regulations
- **Privacy**: Personal data MUST be protected according to privacy laws
- **Security**: Log access MUST be controlled and audited
- **Documentation**: All logging practices MUST be documented for compliance audits
