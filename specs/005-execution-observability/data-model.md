# Data Model: Execution Observability

**Feature**: 005-execution-observability  
**Date**: 2026-03-29  
**Status**: Complete

## Overview

This document defines the data model and entities for the execution observability feature. The feature focuses on logging and monitoring rather than persistent data storage, so the data model primarily represents log entry structures and context information.

## Core Entities

### ExecutionLogEntry

Represents a single execution log event generated by the observability system.

**Purpose**: Captures the lifecycle of method execution across controllers and services

**Fields**:
- `timestamp`: Instant - When the log entry was created
- `executionPhase`: ExecutionPhase - START or COMPLETION
- `componentType`: ComponentType - CONTROLLER or SERVICE
- `componentIdentifier`: String - Fully qualified class and method name
- `correlationId`: String - Unique identifier for the request lifecycle
- `httpMethod`: String - HTTP method (for controllers only)
- `requestPath`: String - Request path (for controllers only)
- `executionDuration`: Long - Duration in milliseconds (for completion entries)
- `executionStatus`: ExecutionStatus - SUCCESS, FAILURE, or ERROR
- `errorCode`: String - Domain error code (for error cases)
- `threadName`: String - Name of the executing thread
- `mdcContext`: Map<String, String> - Complete MDC context at time of logging

**Validation Rules**:
- `timestamp` must not be null
- `executionPhase` must not be null
- `componentType` must not be null
- `componentIdentifier` must not be null or empty
- `correlationId` must not be null or empty
- `executionDuration` must be positive for completion entries
- `executionStatus` must not be null

## Enums

### ExecutionPhase

Represents the phase of method execution being logged.

**Values**:
- `START`: Method execution has begun
- `COMPLETION`: Method execution has finished (success or failure)

### ComponentType

Represents the type of component being logged.

**Values**:
- `CONTROLLER`: REST controller method
- `SERVICE`: Application service method

### ExecutionStatus

Represents the outcome of method execution.

**Values**:
- `SUCCESS`: Method completed successfully
- `FAILURE`: Method failed with an exception
- `ERROR`: System error during logging (rare)

## Value Objects

### MethodIdentifier

Represents the identification of a method for logging purposes.

**Fields**:
- `className`: String - Fully qualified class name
- `methodName`: String - Method name
- `parameterTypes`: List<String> - Parameter type names for method overloading disambiguation

**Validation Rules**:
- `className` must not be null or empty
- `methodName` must not be null or empty
- `parameterTypes` must not be null (can be empty)

### ExecutionDuration

Represents the duration of method execution with validation.

**Fields**:
- `durationNanos`: Long - Duration in nanoseconds
- `durationMillis`: Long - Duration in milliseconds (rounded)

**Validation Rules**:
- `durationNanos` must be non-negative
- `durationMillis` must be non-negative
- `durationMillis` must be consistent with `durationNanos`

## Context Information

### MDCContext

Represents the MDC context information that flows through the system.

**Standard Fields** (as defined in constitution):
- `correlationId`: String - Unique request identifier
- `httpMethod`: String - HTTP method of the request
- `requestPath`: String - Requested API path
- `errorCode`: String - Domain error code when applicable

**Additional Fields**:
- `threadName`: String - Current thread name
- `userId`: String - User identifier (when available)
- `sessionId`: String - Session identifier (when available)

**Validation Rules**:
- `correlationId` must not be null or empty when present
- All field names must be non-null strings
- All field values must be non-null strings

## Log Entry Relationships

### Request Lifecycle Flow

```
Request Start (Controller START)
├── Controller Method Execution
│   ├── Service Method 1 START
│   │   ├── Service Method 1 COMPLETION
│   ├── Service Method 2 START
│   │   ├── Service Method 2 COMPLETION
├── Controller COMPLETION
```

### Correlation Pattern

All log entries within a single request share the same `correlationId`:
- `correlationId`: "req-12345-abcde" (consistent across all entries)
- `executionPhase`: START/COMPLETION (varies by entry)
- `componentType`: CONTROLLER/SERVICE (varies by entry)
- `componentIdentifier`: varies by method being executed

## Data Flow

### Input Data Sources

1. **Method Interception**: AOP aspects provide method signatures and execution timing
2. **MDC Context**: Spring's MDC provides contextual information
3. **Exception Handling**: Exception details from error scenarios
4. **Request Information**: HTTP details from servlet request

## State Transitions

### Method Execution States

```
START → SUCCESS (normal completion)
START → FAILURE (exception thrown)
START → ERROR (logging system error)
```

### MDC Context Lifecycle

```
Request Start → MDC Initialized
Method Execution → MDC Propagated
Error Occurs → errorCode Added
Request End → MDC Cleared
```

## Data Integrity Constraints

### Uniqueness Constraints

- No uniqueness constraints apply to log entries (high-volume append-only data)

### Referential Integrity

- `correlationId` references the logical request lifecycle
- `errorCode` references domain error codes when applicable

### Temporal Constraints

- `timestamp` must be chronologically consistent within request lifecycle
- `executionDuration` must be non-negative and reasonable for the operation

## Performance Considerations

### Data Volume

- Estimated log entries: 2 per method (START + COMPLETION)
- Estimated volume: ~1000 entries/second under normal load
- Storage strategy: Append-only with rotation based on time/size

### Memory Usage

- Log entry objects are short-lived (created, serialized, discarded)
- MDC context is thread-local and reused
- Minimal object allocation per log entry

## Security Considerations

### Sensitive Data Exclusion

The following data is explicitly excluded from log entries:
- Method parameters and return values
- Request body content
- Authentication tokens or credentials
- Personal identifiable information (PII)

### Data Classification

- Log entries are classified as operational data
- Access restricted to authorized personnel
- Retention policies apply based on compliance requirements

## Integration Points

### Existing System Components

1. **CorrelationIdFilter**: Provides initial MDC context setup
2. **GlobalExceptionHandler**: Provides error context and error codes
3. **Logback Configuration**: Provides structured logging output format

### External Systems

1. **Log Aggregation**: Structured JSON format enables easy integration
2. **Monitoring Systems**: Timing data supports performance monitoring
3. **Alerting Systems**: Error status enables automated alerting

## Validation Rules Summary

### Required Fields
- `timestamp`: Must be present and valid
- `executionPhase`: Must be valid enum value
- `componentType`: Must be valid enum value
- `componentIdentifier`: Must be non-empty string
- `correlationId`: Must be non-empty string
- `executionStatus`: Must be valid enum value

### Conditional Fields
- `httpMethod`: Required for CONTROLLER entries
- `requestPath`: Required for CONTROLLER entries
- `executionDuration`: Required for COMPLETION entries
- `errorCode`: Required for FAILURE entries

### Format Constraints
- All string fields must be valid UTF-8
- Timestamp must be ISO-8601 format
- Duration must be non-negative integer
- Component identifiers must follow Java naming conventions
